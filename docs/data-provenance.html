SandLabs — Data Provenance & Datamining Gaps

Where every piece of the served data actually comes from, and exactly what the new file-datamining approach does not cover. As of 2026-06-18, branch feat/monorepo-static-foundation.

Two different questions, don't conflate them:
(1) Is it from game files? — Most of it is game-scraped in origin: item names, combat stats, recipes, loot caps all came out of the files (via the Python extractors) and were baked into the baseline.
(2) Does the current TS transform re-derive it each run? — Only a thin slice: item rarity/icon/description, trampler health, and loot links. The transform consumes just 4 of the 15 datamine datasets, so everything else rides the baseline and won't auto-update per patch until it's wired in. That "wired in" gap — not "is it from game files" — is the real story.

The three layers

LayerWhat it isStatus
Database (Neon/Postgres) The legacy source of truth — full unified entity model (items, parts, tech, locations, links). Exported to the baseline JSON at Foundation. The app no longer reads it. Frozen at Foundation. Never reseeded (hard rule). Now diverges from the served data by every datamine merge since.
Scraped (sek-out/*.json) 15 datasets extracted from game files by the Python scripts. Mostly playtest-era — only compartment_stats.json was regenerated from the release build this session.
Baseline (packages/data/generated/*.json) What the wiki actually serves. = DB export + accumulated datamine merges. The transform reads it, refreshes a subset, writes it back. Live. 378 entities · 39 recipes · 1080 links.

DB vs baseline difference = exactly the datamine merges committed since Foundation: item rarity/icon/description refreshes, the localization enumeration matches (missing-items 48→1), trampler-part health corrections (13 parts), loot-link re-derivation, the disabled Ironclad cargo box, and the renamed radio-beacon boxes. The DB has none of these.

Staleness caveat. Even the “live” fields above are only as fresh as their sek-out/ input. Right now those inputs are mostly playtest-era — only compartment_stats.json (trampler health) was regenerated from the current release files. A full per-patch refresh requires re-running the Python pipeline (build_localization → build_site_data → build_container_loot → …) on the release build, then the transform.

The gap — what we’re missing

Re-derivable from files, just not wired fixable in transform

Extractors already produce these from game files; wiring them into the TS transform would make them auto-update per patch instead of riding the baseline:

  • Item combat stats (itemStats) — weapon_stats.json + turret_stats.json → ~135 items.
  • Authoritative item category/type — item_defs.json (117, extracted this session).
  • Recipes & craft/build costs (the cost role, 233 links) — recipes.json.
  • Location loot caps & contents — locations.json + location_contents.json (27 environments).
  • Item rarity/icon/desc for the ~48 baseline-only items — needs item_defs wired + a fresh items.json rebuild.

Genuinely not in the game files needs other sources

  • Tech tree (techNodeStats + the 536 tech-* links) — server-side (GetResearchTree over WSS). Only the disabled WS-capture addon could get it; currently transcribed.
  • Trampler weight / energy / crew / power — code/runtime-side; stays sand-help baseline.
  • Item & part display names — game uses internal ids; the curated names stay baseline.
  • Translations (i18n) — only EN extracted; needs a multi-locale localization run.
  • Vendor/shop links (buy-*) — no wired extractor; provenance to confirm.

Bottom line: the new approach reliably auto-updates trampler health, loot tables, and item rarity/icon/description (for SEK-covered items) — everything else is either baseline-baked game data waiting to be wired in, or genuinely unobtainable from the files.
